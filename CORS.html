HTTP Security

CORS

Cross-Origin Resource Sharing — everything you need to understand it, once and for all.

What is CORS?
A browser security mechanism that controls which websites can request resources from other domains.
The Same-Origin Policy problem
By default, browsers block scripts on app.com from making requests to api.com. This is called the Same-Origin Policy. CORS is the system that lets servers selectively relax this restriction.
What "origin" means: protocol + domain + port — all three must match. https://app.com:443 and https://api.app.com:443 are different origins — different subdomain.
✓ Same origin
# All same origin as
https://example.com

✓ https://example.com/page
✓ https://example.com/api/v2
✗ Cross origin
# Different origin

✗ http://example.com ← protocol
✗ https://api.example.com ← subdomain
✗ https://example.com:8080 ← port
Key CORS response headers
Access-Control-Allow-Origin: https://app.com # which origins allowed
Access-Control-Allow-Methods: GET, POST, PUT # which methods allowed
Access-Control-Allow-Headers: Authorization # which req headers allowed
Access-Control-Allow-Credentials: true # allow cookies?
Access-Control-Max-Age: 86400 # cache preflight 24h
Common trap: Using Access-Control-Allow-Origin: * with Allow-Credentials: true together is rejected by browsers. For cookies/auth you must specify the exact origin.

1
Browser sends request with Origin header
Browser automatically adds Origin: https://app.com to every cross-origin request. You don't control this — the browser always does it.
GET /data HTTP/1.1 Host: api.com Origin: https://app.com ← auto-added by browser
2
Server processes and responds
The server has no idea CORS exists. It just runs its logic and responds — including whatever Access-Control-* headers you've configured.
HTTP/1.1 200 OK Access-Control-Allow-Origin: https://app.com Content-Type: application/json {"data": "..."}
3
Browser checks the CORS header
The browser reads Access-Control-Allow-Origin and checks: is my origin in there?
If YES → JS gets the response data. Everything works.
If NO → Browser discards the response. JS sees a network error. The server already ran — you can't undo it.
!
The most important thing to remember
CORS errors happen in the browser, not the server. The fix is always on the server side — adding the right response headers. You cannot fix CORS from your frontend code.
Browser vs Postman
CORS is only enforced by browsers — Postman, curl, and server-to-server calls are never blocked.

When does it check?
Not every request triggers a CORS check. The browser is smart about it.
01
Same origin
app.com → app.com
02
Simple cross-origin
GET, HEAD, plain POST
03
Complex cross-origin
DELETE, PUT, custom headers
No CORS check at all
Same origin requests skip CORS entirely. Browser doesn't even think about it — there's no cross-origin risk if the request comes from the same place.
// Both from https://app.com — no CORS check fetch('/api/users') // ✓ same origin, instant fetch('https://app.com/api') // ✓ same origin, instant
Checks on every request — no extra round trip
Simple requests are sent directly. The browser reads the CORS headers in the response and decides whether to pass the data to JS. One round trip only.
// "Simple" = GET, HEAD, or POST with no custom headers fetch('https://api.com/data') // → sends request → checks response headers // 1 round trip every time, but just a header read — basically free
A simple request is safe to fire blindly because even if the server runs it, blocking the response means your JS never sees any data that could be leaked.
Preflight first — then cached
First call sends an OPTIONS preflight. Server responds with what's allowed. Browser caches it. All subsequent calls to the same endpoint skip the OPTIONS.
// First call to DELETE /user OPTIONS /user → Access-Control-Max-Age: 86400 ← preflight DELETE /user → 200 OK ← real request // Next 24 hours — no OPTIONS sent DELETE /user → 200 OK // preflight cached ✓
Why preflight?
The browser won't fire a destructive request without asking permission first.

The core reason
For a GET, blocking the response is enough. The server didn't change anything.

But for DELETE /account — the server executes it the moment it arrives. By the time the browser sees the response and thinks "wait, this origin isn't allowed", your account is already deleted. Blocking the response at that point is useless.
Preflight is asking permission before pulling the trigger. OPTIONS is used because it's defined in the HTTP spec as safe and side-effect free — perfect for a "will you allow this?" knock.
WITHOUT PREFLIGHT
evil.com fires DELETE /account
Server executes it ← account gone
Browser blocks response ← useless
Damage already done.
WITH PREFLIGHT
Browser sends OPTIONS first
Server: "evil.com? Not allowed."
Browser stops. DELETE never sent.
Account safe. ✓
Summary
Everything we covered, condensed.
Mental model
Browser = the bouncer. It enforces the rules.
Server = the guest list. It posts who's allowed.
CORS = not your API's security. It protects users from malicious pages.
Fix is always server-side. You can't fix CORS in frontend code.
Concept What it means
Same-Origin Policy Default browser rule — scripts can only talk to their own origin
CORS System for servers to opt-in to cross-origin requests
Simple request GET/HEAD/plain POST — sent directly, response checked
Preflight OPTIONS request sent before risky calls — permission check
Max-Age Caches preflight result — avoids OPTIONS on every call
Postman bypass No browser = no CORS — Postman/curl always work regardless
