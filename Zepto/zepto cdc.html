Source DB
STEP 01
Postgres — Source of Truth
Zepto's transactional database stores orders, inventory, and state transitions. An order like id=123 can go through STATE_A → B → C → D in milliseconds during peak traffic. Every row-level change (INSERT / UPDATE / DELETE) is first written to the WAL (Write Ahead Log) before hitting the actual table.
-- Order 123 changes 4 times in milliseconds
UPDATE orders SET state = 'STATE_A' WHERE id = 123; -- 10:00:00.001
UPDATE orders SET state = 'STATE_B' WHERE id = 123; -- 10:00:00.002
UPDATE orders SET state = 'STATE_C' WHERE id = 123; -- 10:00:00.003
UPDATE orders SET state = 'STATE_D' WHERE id = 123; -- 10:00:00.004

STEP 02
Debezium — CDC Engine
Debezium reads the Postgres WAL like a live feed — it sees every change in order, without touching actual tables. It converts each WAL entry into a structured change event and publishes it to Kafka. Zero impact on the source DB's query performance.
CHANGE EVENT (Kafka message)
{ op: "u", // update
  before: { state: "STATE_A" },
  after: { state: "STATE_B" },
  ts_ms: 1700000000002 }

STEP 03
Kafka — Event Bus
All 4 change events for id=123 land in the same Kafka topic partition, in order. The JDBC Sink Connector polls Kafka and groups these into a batch to write to Postgres. During peak traffic, batches can contain hundreds of events — many of them for the same order ID.
PARTITION: id=123 STATE_A → id=123 STATE_B → id=123 STATE_C → id=123 STATE_D → id=456 STATE_A … batched together

Problem
STEP 04 — THE PROBLEM
reWriteBatchedInserts=true + Duplicate Keys = CRASH
The JDBC driver flag reWriteBatchedInserts=true merges all inserts in a batch into a single multi-row SQL statement for performance. But when the same primary key appears more than once (STATE_A and STATE_D for id=123), Postgres cannot resolve which value to use in the ON CONFLICT DO UPDATE clause — and refuses entirely.
❌ Without Dedup — CRASHES
INSERT INTO orders VALUES
  (123, 'STATE_A'),
  (123, 'STATE_B'), ← duplicate!
  (123, 'STATE_C'), ← duplicate!
  (123, 'STATE_D') ← duplicate!
ON CONFLICT (id) DO UPDATE...

ERROR: ON CONFLICT DO UPDATE command
cannot affect row a second time.
⚠️ Root Cause
Postgres evaluates ON CONFLICT against the original table — not row by row within the VALUES list.

When it sees id=123 four times, it has no way to know which value should "win" — so it rejects the entire statement.
STEP 04b — WRITE AMPLIFICATION
Even Without Crash: HOT Updates Fail → Index Write Explosion
Even when not crashing, rapid state changes on the same row cause Postgres's HOT (Heap Only Tuple) optimization to fail. HOT skips index updates if the updated column has no index AND the new row fits on the same heap page. Since state is indexed (you query by state constantly), HOT never applies — every update rewrites all indexes.
4 STATE CHANGES = 4x WRITES
STATE_A write:
  1 heap write
  + N index writes
STATE_B write:
  1 heap write
  + N index writes
STATE_C, D... same

Total: 4 heap + 4N index writes
for 1 logical change!
HOT CONDITIONS (both must be true)
✓ No indexed column changed
✓ New row fits on same page

In CDC workloads:
✗ state column IS indexed
✗ Pages fill up from rapid writes

→ HOT never applies

Impact
STEP 05 — PEAK TRAFFIC IMPACT
AAS (Average Active Sessions) Explodes on COMMIT
During peak traffic, hundreds of write batches hit Postgres simultaneously. Every batch ends with a COMMIT — which forces a disk flush (WAL write). Disk I/O becomes the bottleneck. Sessions pile up waiting to commit → AAS spikes → death spiral.
Normal Time
AAS = 1 (sessions completing fast)
COMMIT → done in ~1ms → session frees up. Disk keeps up easily.
Peak Traffic
AAS = 50-200+ (sessions piling up)
COMMIT → waiting 50ms → 100ms → more arrive → disk overwhelmed → death spiral 💀
the fix
The Fix
STEP 06 — REDUCTION BUFFER
In-Memory Deduplication Before Flush
The Reduction Buffer sits between Kafka and Postgres. Before flushing any batch to the database, it groups events by primary key and keeps only the latest event per ID. Intermediate states are discarded — the database only ever sees the final truth.
BATCH IN (6 events)
id=123 STATE_A → id=123 STATE_B → id=123 STATE_C → id=123 STATE_D ✓ | id=456 STATE_A ✓ | id=789 STATE_B ✓
AFTER DEDUP — ONLY 3 ROWS TO POSTGRES
INSERT INTO orders VALUES
  (123, 'STATE_D'), ✓ no duplicates
  (456, 'STATE_A'),
  (789, 'STATE_B')
ON CONFLICT (id) DO UPDATE SET state = EXCLUDED.state;
