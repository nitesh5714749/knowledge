02 · The Protocol

MCP — the USB-C port for AI

Model Context Protocol is an open standard that lets AI clients connect to third-party services via a simple config file. You point your AI client at an MCP server and it gains a set of callable tools. Zepto built exactly this — a shopping MCP server that exposes their catalogue, cart, and checkout as tools the AI can invoke.

03 · The Hard Part

OAuth 2.1 + PKCE — why it matters

Shopping needs a logged-in user. But AI agents are "public clients" — unlike a backend server, they can't securely store a client secret. Standard OAuth would break here. Zepto solved this with OAuth 2.1 and mandatory PKCE (Proof Key for Code Exchange).

The problem

AI agents can't keep secrets

Traditional OAuth needs a client_secret stored server-side. AI clients run in user environments — memory can be inspected, network traffic intercepted. A stored secret is a liability.

The solution

PKCE — a one-time cryptographic proof

Generate a random secret and its hash at the start of each login. The secret never travels over the network. Only the hash does. At exchange time, prove you had the original.

Step by step — the full PKCE flow
1
AI Client — internal

Generate a secret pair, locally in memory

Before touching Zepto's servers, the AI client generates two linked values entirely on-device. The raw verifier never leaves the client in plaintext.

// Generated locally, stored only in memory code_verifier = "xK9mTq2rZ8pL..." // random 32-byte string code_challenge = SHA256(code_verifier) // = "7rF4aZ1..." // Only the hash (challenge) is safe to send over the network
2
AI Client → Zepto Auth Portal

Send the challenge (hash), not the secret

The client starts the OAuth flow by sending the hashed challenge to Zepto's Auth Portal. Even if intercepted, the attacker only gets the hash — useless without the original verifier.

GET /oauth/authorize ?code_challenge = "7rF4aZ1..." &code_challenge_method = "S256" &client_id = "zepto-mcp-client" &redirect_uri = "mcp://callback" &scope = "cart checkout"
💾

Zepto stores code_challenge = "7rF4aZ1..." against this session. Used to verify the client in Step 6.

3
Zepto Auth Portal ← User

User authenticates via OTP

Zepto redirects the user to its Auth Portal (a Next.js app). The user verifies via phone OTP — Zepto's existing identity infrastructure. On success, a short-lived authorization_code is issued.

// Auth Portal flow 1. Zepto sends OTP → user's registered phone 2. User enters OTP in the portal 3. OTP validates → issues: authorization_code = "auth_9xP...k3T" // Valid for ~60 seconds, single-use only
4
Zepto → AI Client (via redirect)

Authorization code delivered — the intercept-risk moment

Zepto redirects the browser to the AI client's callback URI with the authorization code. This redirect is the riskiest point — it can be intercepted by a malicious app that registered the same URI scheme.

mcp://callback?code="auth_9xP...k3T"
⚠️

An attacker intercepts this redirect and steals "auth_9xP...k3T". Without PKCE they'd exchange it for an access token instantly. With PKCE, they're about to hit a wall.

5
AI Client → Zepto Token Endpoint

Exchange code + verifier — the PKCE moment

To get the actual access token, the client must send both the authorization code AND the original code_verifier. The attacker cannot do this — they only have the hash, not the original.

// Legitimate AI client POST /oauth/token code = "auth_9xP...k3T" code_verifier = "xK9mTq2rZ8pL..." ← the original secret grant_type = "authorization_code" // Attacker's attempt POST /oauth/token code = "auth_9xP...k3T" ← stolen ✓ code_verifier = ??? ← no idea ✗
⛔

The attacker is completely stuck. SHA256(anything they guess) won't equal "7rF4aZ1...". Brute-forcing a 32-byte random value is computationally infeasible.

6
Zepto Auth Server — verification

Re-hash and verify — then issue tokens

Zepto runs SHA-256 on the incoming code_verifier and compares it to the stored code_challenge. If they match, the client proved it's the same one that started the flow.

SHA256("xK9mTq2rZ8pL...") = "7rF4aZ1..." stored_challenge = "7rF4aZ1..." ✓ Match — tokens issued! { "access_token": "eyJ...", // short-lived JWT "refresh_token": "rft_...", // longer-lived "expires_in": 3600 }
✅

Attacker's request returns 400 invalid_grant. The stolen code is also now invalidated — it's single-use. They get nothing.

7
AI Client — shopping begins

Access token in hand — full Zepto unlocked

The AI client now holds tokens it uses to call Zepto's MCP tools on your behalf. Every API call carries the JWT in the Authorization header.

Authorization: Bearer eyJ... search("cheese for a dinner party") add_to_cart(sku="ZP-9821", qty=2) checkout() // places the real order

Why a separate Auth Portal in Next.js?

The Auth Portal is a standalone Next.js app, deliberately separate from the MCP server. This lets Zepto reuse their existing Legacy OTP identity infrastructure without rebuilding it. The MCP server stays stateless — it just says "go talk to the Auth Portal."

It became a generic OAuth socket — any future Zepto service needing user auth can plug into it without rebuilding the entire flow. One auth system for everything.

05 · After Login

Yes — the AI stores tokens. Here's why that's fine.

After the OAuth flow, the AI client holds an access token and a refresh token. Unlike the authorization_code that PKCE protects during acquisition, tokens are designed to be client-held — but they come with properties that limit the damage if they're ever compromised.

Why tokens are safer than passwords
01

Scoped, not omnipotent

A leaked access token only grants cart + checkout on Zepto. A leaked password grants everything, everywhere it's reused. Scoping is the single biggest practical mitigation.

Scoping
02

Short-lived by design

Access tokens expire in ~1 hour. An attacker has a narrow window — and after expiry it's completely useless with no way to renew without the refresh token.

Expiry
03

Server-side revocation

Zepto can invalidate any refresh token instantly — on user logout, a reported compromise, or suspicious usage patterns. No password reset required.

Revocability
04

Stored in OS secure storage

MCP clients store refresh tokens in the OS keychain (macOS Keychain, Windows Credential Manager) — not in plaintext files or localStorage.

Storage
05

Refresh token rotation

Each time a refresh token is used, a new one is issued and the old one is invalidated. If a leaked token is used by an attacker, the real client's next refresh detects the conflict automatically.

Rotation
The key distinction

PKCE protects the acquisition of tokens. Token properties — scope, expiry, revocability — protect against the tokens themselves being compromised. These are two separate threat surfaces, and Zepto covered both.

An attacker who intercepts the authorization_code redirect can't get tokens without the verifier. An attacker who somehow steals an access token can only use it for scoped actions, for a short window. Defense in layers — not a single lock.
