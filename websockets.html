WebSocket Connections Deep Dive

Understanding state, capacity, and how to estimate how many connections your server can handle.

Core Concept

An Open Socket Mostly Consumes State

Most engineers instinctively fear large WebSocket numbers because they compare them to HTTP requests under load. That comparison is wrong. An open socket is not an active request — it's a row of state sitting in memory, waiting silently.

Key insight: 100,000 idle WebSocket connections do not mean 100,000 things happening simultaneously. The CPU is mostly sleeping. The server is just remembering those connections exist.
📁

File Descriptors

Each connection gets a slot (an integer) in the OS file descriptor table. Configurable up to ~500K+.

🧠

Kernel Buffers

The OS allocates small send/receive buffers (~4–8 KB each) per socket to stage data.

🔐

TLS State

Secure connections keep session keys and cipher context alive — ~50–100 KB of memory.

👤

App Metadata

Your code stores which user owns this socket, which rooms they joined, timestamps, etc.

⚡

No Constant CPU

While idle, zero CPU cycles are spent on the connection. The kernel simply holds the state.

🪣

Small App Overhead

Framework socket objects (e.g. Node.js V8 heap) cost ~2–4 KB per connection.

Architecture

Both Client and Server Hold State

A WebSocket connection is a two-way street. Both endpoints — the browser and your server — allocate and hold state for the lifetime of the connection.

🖥️ Server Side

Holds state for every connected client. With 100K clients, the server holds 100K file descriptors, 100K TLS contexts, 100K buffer pairs, and 100K metadata records. This is where capacity planning matters.

💻 Client Side

Holds state for just its own connection (usually one). The cost is identical per-connection, but since a browser only opens 1–2 sockets, the client never feels it.

📞 Analogy: Phone Call

Both your phone and the other person's phone are "on the call" — both hold state. But the phone exchange (the server) is the one managing all calls at once. Your phone cares about 1 call; the exchange juggles millions. Same resource cost per connection, completely asymmetric burden.

Comparison

WebSockets vs HTTP APIs Under Load

⚡ HTTP API (100K req/sec)

Every request demands CPU right now — parse headers, query DB, serialize JSON, send response, tear down. 100K req/sec = 100K units of active CPU work per second. Like a restaurant with no reservations: every customer is served immediately.

🔌 WebSocket (100K connections)

Connections sit open silently. CPU only wakes up when a message arrives. If 5% are active at once, you're handling 5K events/sec — not 100K. Like a hotel: 100K guests checked in, but most are sleeping. Staff only responds when a guest rings.

The C10K lesson: Old servers used one thread per connection — 10K threads meant real CPU overhead (context switching). Modern event-driven servers (Node.js, Go, Nginx) use epoll/kqueue — the OS only wakes the app when a socket has data. One thread can serve hundreds of thousands of idle sockets.
Capacity Planning

How to Estimate Connection Capacity

The formula is simple: figure out what each connection costs, then divide your total available resource by that cost. Do this for every dimension.

Example machine: 8 CPU cores · 16 GB RAM · 1 Gbps NIC

Memory Per Connection

60 KB
📥 Recv Buffer
8 KB
📤 Send Buffer
8 KB
👤 App Metadata

~100 KB
// Memory calculation
Total RAM      = 16 GB  =  16,384 MB
OS + App       =  2 GB  (reserved)
Available      = 14 GB  =  14,680,064 KB

Max connections = 14,680,064 KB ÷ 100 KB
               ≈ 146,000 connections

All Dimensions at a Glance

Resource Available Per Connection Max Connections
🧠 RAM 14 GB usable ~100 KB ~140,000 ⬅ binding
📁 File Descriptors 500K (after tuning) 1 500,000
🌐 Network (1 Gbps) 1,000 Mbps ~0.02 Mbps (light) >10,000,000
⚙️ CPU (8 cores) 8,000 ms/sec ~0.005 ms/sec (5% active) >1,000,000
🔌 Server Ports Not a constraint — Unlimited
RAM is almost always the binding constraint. File descriptors need a one-time config change. Network and CPU only become issues at extreme message rates. Focus your math on memory first.
Optimization

Pushing Beyond the Limit

Want more connections on the same hardware? Attack the memory cost per connection.

🔐

Offload TLS to a Load Balancer

Put Nginx or HAProxy in front. Your app handles plain TCP internally, saving ~60 KB per connection — the biggest single win.

🔧

Tune Kernel Socket Buffers

Lower net.core.rmem_default and wmem_default to 4096 bytes. This halves your buffer cost without hurting throughput for low-bandwidth connections.

🗂️

Keep App Metadata Lean

Store only a user ID in memory. Fetch the full user record from Redis or a cache on demand. Don't embed entire objects per socket.

📁

Raise the File Descriptor Limit

Set ulimit -n 500000 in your systemd unit or /etc/security/limits.conf. Default of 1024 will kill you at small scale.

📊

Always Load-Test, Don't Trust Theory Alone

Use tools like k6, artillery, or wrk to verify. Theory shows you what to watch; the benchmark shows when it breaks.
