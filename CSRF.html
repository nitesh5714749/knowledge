CSRF — Cross-Site Request Forgery

A malicious site tricks your browser into making a request to another site where you're already logged in — using your session cookie without your knowledge.

attacker goal: DO something uses your cookie
CORS — Cross-Origin Resource Sharing

A browser policy that controls which origins can read responses from another origin. Prevents malicious JS from stealing data from other sites.

attacker goal: READ something blocks response reading
Key insight CORS and CSRF are not alternatives — they guard against different threats. A secure app needs both. CORS stops data theft; CSRF tokens stop unwanted actions.

The CSRF Attack

How a malicious website silently performs actions on your behalf using your browser's cookies.

The root problem Browsers automatically attach cookies to every request — regardless of which site triggered the request. The server can't tell if you intended the request or were tricked into it.
Step-by-step attack flow
1

You log into bank.com

Server sets a session cookie: sessionid=abc123 in your browser.

2

You visit evil.com (in another tab)

The attacker's page loads silently. You see a cat picture. Behind the scenes, hidden HTML is loaded.

3

Hidden request fires automatically

<!-- invisible on the page --> <img src="https://bank.com/transfer?to=hacker&amount=5000" width="0" height="0">

The browser makes a GET request to bank.com.

4

Browser auto-attaches your cookie

Because you're logged in, the browser appends Cookie: sessionid=abc123 to the request.

5

Bank server executes the transfer

It sees a valid session cookie. It has no idea you didn't mean to do this. $5000 is gone.

A more dangerous form-based variant

POST requests can be forged too — with an auto-submitting form:

<!-- evil.com hides this and auto-submits on page load --> <form id="x" method="POST" action="https://bank.com/transfer"> <input name="to" value="hacker"> <input name="amount" value="5000"> </form> <script>document.getElementById('x').submit()</script>

CSRF Defense

How the CSRF token pattern stops forged requests — and how frameworks handle it automatically.

The fix: prove the request came from YOUR page Include a secret token in every state-changing request. The evil site can't read your page, so it can't know the token. No token = rejected.
Protected flow
1

Server generates a random token

Tied to your session. Stored server-side. Cryptographically unpredictable.

2

Token embedded in the HTML form

<form method="POST" action="/transfer"> <input type="hidden" name="_csrf" value="xK9p2mQr7vL4nB..."> <!-- your real fields --> </form>
3

User submits — token travels in the body

The token is sent as form data (not a cookie), alongside the session cookie.

4

Server compares: body token == session token?

Match → allow. Mismatch / missing → 403 Forbidden.

Framework shortcuts — you don't write this manually

Django

<form method="POST"> {% csrf_token %} <!-- that's it --> </form>

Laravel / Blade

<form method="POST"> @csrf <!-- done --> </form>
For AJAX / fetch calls

No form = no auto-injection. Read the token from a meta tag and send it as a header:

// Server puts token in <meta name="csrf-token" content="..."> const token = document.querySelector('meta[name="csrf-token"]').content; fetch('/transfer', { method: 'POST', headers: { 'X-CSRF-Token': token, // <-- evil.com can't set this 'Content-Type': 'application/json' }, body: JSON.stringify({ to: 'alice', amount: 100 }) });
Why a custom header blocks evil.com Custom headers like X-CSRF-Token trigger a CORS preflight — only same-origin JS can set them. A forged form or img tag physically cannot set custom headers.

Why CORS doesn't prevent CSRF

The most common misconception in web security — they look related but solve entirely different problems.

CORS blocks

Malicious JS reading the response from another origin.

fetch('bank.com/balance')
→ response BLOCKED by CORS
CSRF exploits

The attacker doesn't need to read anything. Damage done when request executes.

<img src="bank.com/transfer">
→ request SENT, CORS irrelevant
The critical gap <img>, <form>, and <script> tags trigger "simple requests" that bypass CORS preflight entirely. The request fires. The cookie goes with it. CORS never intervenes.
Comparison table
Property CORS CSRF token
Threat type Data theft (reading) Unwanted actions (writing)
Attacker needs response? Yes — wants data back No — just needs action to fire
Where enforced Browser (headers) Server (validates token)
Stops img/form attacks? No Yes
Stops JS data theft? Yes No
Needed together? Yes — they guard different things

Token Deep Dive

Where the token lives, why it can't be stolen, and the double-submit cookie variant.

Session cookie

Lives in the browser cookie jar. Sent automatically on every request to that domain — always, no exceptions.

auto-sent by browser

CSRF token

Lives in the HTML body or a JS variable. Only sent when your form submits or your code explicitly includes it.

requires page access to read
Why can't evil.com just grab the token?

The Same-Origin Policy prevents a script on evil.com from reading the HTML or cookies of bank.com. Even if evil.com fires a request to bank.com, it cannot read what comes back. So the token, which lives inside bank.com's HTML response, is invisible to the attacker.

Variant: Double-submit cookie pattern

Used by stateless servers that don't want session storage. The token IS in a cookie — but the server checks that it's also echoed in the request body:

// Server sets a readable cookie (NOT HttpOnly) Set-Cookie: csrf_token=xK9p2mQr7; SameSite=Strict // Your JS reads it and echoes it in the request body const csrfToken = getCookie('csrf_token'); fetch('/transfer', { body: JSON.stringify({ csrf_token: csrfToken, ... }) }); // Server checks: cookie value == body value?
Why evil.com still can't forge this The cookie is sent automatically — but evil.com can't read the cookie value (same-origin policy). It can't echo a value it doesn't know.
Modern alternative: SameSite cookie
Set-Cookie: sessionid=abc123; SameSite=Strict; HttpOnly; Secure

With SameSite=Strict, the browser simply refuses to attach the cookie to any cross-site request. No token needed. Most modern browsers support this — but CSRF tokens remain best practice for defense in depth.

SameSite valueBehavior
StrictCookie never sent cross-site. Strongest protection.
LaxCookie sent on top-level navigations only (GET). Default in modern browsers.
NoneCookie always sent. Requires Secure flag. Legacy behavior.
