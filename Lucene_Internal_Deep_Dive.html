01 Inverted Index

At the core of every search engine is an inverted index — a lookup table mapping each term to the list of documents that contain it. Click a term to highlight its documents.

⚠️
The scaling problem. In a real search engine, common terms like "the" or "is" can appear in millions of documents. A posting list of [1, 100000, 200000, … 9999999] stored as plain 32-bit integers would cost ~40 MB just for one term. Lucene needs a smarter approach.
02 Delta Encoding

Instead of storing raw document IDs, Lucene stores the gaps (deltas) between consecutive IDs. Since IDs are sorted and the gaps are small, the numbers become much smaller — and smaller numbers need fewer bits.

// Original IDs (sorted)
raw = [10, 40, 50, 90]

// Delta encoding rule
delta[0] = raw[0] // = 10 (anchor)
delta[i] = raw[i] - raw[i-1] // gap from previous

// Result
delta = [10, 30, 10, 40]

// Decode: prefix sum restores original
decoded[i] = Σ delta[0..i]
💡
Why this helps: The max value dropped from 90 → 40. A value of 90 needs 7 bits. A value of 40 needs 6 bits. At scale, going from needing 27 bits per ID to 6 bits is a 4.5× compression.
03 Blocking for Efficiency

Lucene doesn't compress the entire list in one shot. It splits the delta-encoded list into fixed-size blocks (called PackedDocDeltaBlock). Each block is independently compressed based on the maximum value it contains.

// delta list split into 2 blocks of size 2

🔑
Why blocks? If one outlier value of 10,000,000 were in the list, using the whole list's max would force every value to use 24 bits. With blocks, only the block containing that outlier pays the high bit-cost. Other blocks stay cheap.
