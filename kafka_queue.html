01The problem with consumer groups

Kafka's normal consumer groups assign partitions exclusively — one partition to exactly one consumer. This causes two fundamental problems for job queue use cases.

-- Normal consumer group (3 consumers, 1 partition)
Topic: [P0]   [P1]   [P2]
         |      |      |
        C1     C2     C3   ← each consumer owns exactly one partition

-- What if you only have 1 partition?
Topic: [P0]
         |
        C1   C2 (idle)   C3 (idle)   ← 2 consumers doing nothing
Over-partitioning problem: To get 10 workers processing a topic in parallel, you need at least 10 partitions. Changing partition count is expensive and risky on live topics.

Max consumers = partition countCan't scale consumers independently
Offset-based ack onlyCan't retry a single failed message

No per-message delivery trackingNo poison message detection

02Share groups — the solution

KIP-932 introduces share groups — a new kind of consumer group where multiple consumers can pull from the same partition simultaneously. The partition is no longer exclusively owned.

-- Share group (5 consumers, 1 partition)
Topic: [P0]
         |
   ┌─────┼─────┐
  C1   C2   C3   C4   C5   ← all 5 consumers get work

-- Each consumer gets individual messages, not partitions
msg #1 → C1     msg #4 → C4
msg #2 → C2     msg #5 → C5
msg #3 → C3     msg #6 → C1 (finished, picks up next)
Key difference: The share-partition leader on the broker decides which consumer gets which message. Consumers no longer own partitions — they just request work.

Per-message acknowledgement

Instead of committing an offset (marking everything before it as done), each message is acked individually with one of three outcomes:

consumer.acknowledge(record, AcknowledgeType.ACCEPT);   // done ✓
consumer.acknowledge(record, AcknowledgeType.RELEASE);   // give to someone else ↺
consumer.acknowledge(record, AcknowledgeType.REJECT);    // unprocessable, delivery count++ ✕

// acks are batched in memory, sent on:
consumer.commitAsync();

04How the leader tracks state

The share-partition leader is a broker-side component co-located with the partition leader. It maintains an in-memory table of every offset in the active window.

Per-offset state table

// What the leader stores for each offset in the window
{
  offset: 4,
  state: "Acquired",
  consumer_id: "consumer-3",
  delivery_count: 1,
  lock_expiry: now + share.record.lock.duration.ms
}

Four message states

StateMeaningNext state
AvailableReady to be dispatched to a consumerAcquired
AcquiredHeld by a consumer, lock timer runningAcknowledged / Available / Archived
AcknowledgedSuccessfully processedArchived (when SPSO advances)
ArchivedDone — removed from window—

The lock timer — crash safety

Every Acquired message has a timer. If the consumer crashes and never acks, the timer fires and the message automatically returns to Available. The leader doesn't need the consumer to do anything.

// Config: how long a consumer can hold a message
share.record.lock.duration.ms = 30000   // 30 seconds default

// If lock expires:
state: Acquired → Available
delivery_count: 1 → 2

// If delivery_count hits the limit (default 5):
state: Acquired → Archived   // message silently dropped

Durability — the __share_group_state topic

This entire state is periodically checkpointed to an internal Kafka topic __share_group_state via the share coordinator. If the broker crashes, the state is replayed from this topic on restart — no messages are lost.

// State persisted as snapshots + incremental updates
ShareSnapshot  { spso, states[] }       ← full checkpoint
ShareUpdate    { offset, new_state }    ← incremental delta

// On broker restart:
1. Read latest ShareSnapshot from __share_group_state
2. Replay all ShareUpdate records after it
3. Rebuild in-memory state table

05The sliding window — SPSO & SPEO

The share-partition leader manages a sliding window of in-flight offsets bounded by two pointers: SPSO (Share Partition Start Offset) and SPEO (Share Partition End Offset).

SPSO = leftmost offset still being tracked (first unarchived)
SPEO = rightmost offset currently dispatched

Everything before SPSO  → Archived (done, out of window)
Everything after  SPEO  → Not yet fetched
Everything between      → In the window (Available / Acquired / Acknowledged)
The critical rule: SPSO advances to the first offset that is NOT yet resolved. It scans left-to-right and stops at the first Acquired offset. It cannot jump over holes.

Why SPSO can't jump over holes

The leader must archive offsets in order because Kafka's log is sequential. If it archived offset 5 while offset 3 was still in-flight, and offset 3 later needed to be redelivered, the leader would have no way to track it — it's already outside the window. The left-to-right scan guarantees every offset is either resolved or still tracked.

// SPSO advance algorithm (pseudo-code)
function advanceSPSO():
  while stateTable[spso].state in (Acknowledged, Archived):
    stateTable[spso].state = Archived
    spso++
  // stops at first Acquired or Available offset

06Cons & trade-offs

Share groups are powerful but come with real limitations you need to understand before adopting them.

No ordering guarantees
Multiple consumers pull from the same partition. Message 7 may be processed before message 3. Any key-based ordering is gone entirely.
Slow left edge stalls window
One slow or crashed consumer holding the leftmost offset blocks SPSO from advancing — even if all other messages are done. Memory stays allocated for the entire stalled window.
No dead letter queue (4.0/4.1)
Messages that exceed max delivery attempts are silently archived. There is no DLQ to inspect what failed or why. You must build this yourself.
No delayed retries
Release re-dispatches immediately — no "retry in 30s" like SQS. Poison messages can cause rapid spin-loops across all consumers until archived.
Window size cap on parallelism
group.share.partition.max.record.locks caps total in-flight messages. Once hit, the leader stops dispatching until some acks come in.
No fetch-from-follower
All reads must go through the partition leader broker. Rack-aware reads are not supported. Higher cross-rack traffic, more load on one broker.
Broker-side overhead
Per-offset state tracking, lock timers, and checkpointing to __share_group_state all run on the broker. Risk of noisy-neighbor effects on write throughput.
Not production-ready yet
Preview in Kafka 4.1. Record formats were unstable in 4.0 — clusters using share groups in 4.0 cannot be upgraded. Not recommended for production workloads.
When to use share groups: Messages are truly independent work items (no ordering dependency), you want to scale consumers beyond partition count, and you can tolerate at-least-once delivery with no guaranteed ordering.
When NOT to use: Any ordering requirement, any key-based processing dependency, or any use case needing reliable DLQ + delayed retry — use SQS, RabbitMQ, or a normal consumer group with more partitions instead.

Key config knobs

// Broker
group.share.partition.max.record.locks = 200    // max in-flight per partition
share.record.lock.duration.ms          = 30000  // crash recovery window

// Tune lock duration carefully:
//   too long  → long stalls when consumer crashes
//   too short → excessive redeliveries under normal load

// Enable (not for production)
group.coordinator.rebalance.protocols = classic,consumer,share
unstable.api.versions.enable          = true     // required in 4.0
