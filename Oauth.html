OAuth 2.0

A complete interactive breakdown of how delegated authorization works — from actors to tokens to attack vectors.

What is OAuth 2.0?

OAuth 2.0 is an authorization framework — not authentication — that lets a user grant a third-party application limited access to their resources on another service, without sharing their password. Think "Sign in with Google" or a weather app reading your Google Calendar.

⚠️ Authorization ≠ Authentication

OAuth 2.0 proves what you can access, not who you are. For identity (login), you need OpenID Connect (OIDC), which is a thin identity layer built on top of OAuth 2.0.

The Four Actors

👤
Resource Owner
The User
The human (or system) who owns the data and grants permission. They approve what the app can access.
📱
Client
The App
The application requesting access. Could be a web app, mobile app, or backend service. It holds a client_id and client_secret.
🏛️
Authorization Server
Auth Server
Issues tokens after verifying the user's identity and consent. Examples: Google, GitHub, Okta, Auth0, Keycloak.
🗄️
Resource Server
The API
Hosts the protected resources (e.g. Gmail API, GitHub API). Validates tokens on every request before serving data.
The Core Analogy — Hotel Key Card 🏨

You (user) check in at the front desk (auth server) and get a key card (access token). The key card only opens certain doors (scopes) and expires at checkout. You never give your master key (password) to the valet (third-party app) — you give them a limited valet key instead.

Authorization Code Flow + PKCE
The most secure and recommended flow

"Login"
User Initiates Login
The user clicks "Login with Google" in your app. The app generates a random state parameter (CSRF protection) and a code_verifier + code_challenge for PKCE.
// App generates PKCE values code_verifier = random_43-128_char_string code_challenge = BASE64URL(SHA256(code_verifier)) state = random_opaque_value
2
Redirect to
Auth Server
Authorization Request
The app redirects the browser to the auth server's /authorize endpoint with all required parameters.
GET https://accounts.google.com/o/oauth2/v2/auth ?response_type=code &client_id=your_app_client_id &redirect_uri=https://yourapp.com/callback &scope=openid email profile &state=xK9mN2pQ &code_challenge=E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM &code_challenge_method=S256
3
User Logs In
& Consents
Authentication & Consent
The auth server shows a login page. The user authenticates (enters password, 2FA, etc.) and sees a consent screen listing exactly what the app wants to access. The user clicks "Allow".
// Auth server shows consent screen: // "YourApp wants to access:" // ✓ View your email address // ✓ See your personal info // ✓ See your primary Google Account email // // [Allow] [Cancel]
4
Auth Code
Returned
Authorization Code Issued
Auth server redirects back to your app's redirect_uri with a short-lived authorization code (valid ~60s). The app verifies the state matches what it sent.
GET https://yourapp.com/callback ?code=4/0AY0e-g7... ← short-lived! &state=xK9mN2pQ ← must match!
5
Exchange
Code → Token
Token Exchange (Server-to-Server)
Your backend POSTs the authorization code to the token endpoint, along with the original code_verifier (PKCE). The auth server verifies the hash matches and returns tokens. This happens server-side — never in the browser.
POST https://oauth2.googleapis.com/token grant_type=authorization_code code=4/0AY0e-g7... redirect_uri=https://yourapp.com/callback client_id=your_app_client_id client_secret=YOUR_SECRET ← server-side only code_verifier=original_verifier ← PKCE
6
Tokens
Issued 🎉
Access & Refresh Tokens
The auth server responds with tokens. The access_token is used for API calls. The refresh_token gets new access tokens when they expire.
// Response body (JSON): { "access_token": "ya29.a0AfH6SM...", "token_type": "Bearer", "expires_in": 3600, ← 1 hour "refresh_token": "1//0eXm8...", ← long-lived "scope": "openid email profile", "id_token": "eyJhbGci..." ← OIDC only }
7
Access
Resources
API Call with Token
For every API call, the app sends the access token in the Authorization header. The resource server validates the token and returns data.
GET https://www.googleapis.com/oauth2/v1/userinfo Authorization: Bearer ya29.a0AfH6SM... Accept: application/json // Response: { "id": "1234567890", "email": "user@gmail.com" }

What is a Grant Type?

A grant type defines how the client obtains tokens. Different situations (user present, machine-to-machine, legacy apps) require different flows. Choose the right one for your use case.

Authorization Code + PKCE
Web apps, Mobile apps, SPAs
The gold standard. User-facing apps where a human authorizes. Always use with PKCE now.
Client Credentials
Machine-to-Machine (M2M)
No user involved. Backend services authenticating with each other. Uses client_id + client_secret directly.
Device Authorization
Smart TVs, CLIs, IoT
For input-constrained devices. Shows a code the user types on another device (e.g. your phone).
Refresh Token
Renew expired access tokens
Not a standalone flow — used to silently get new access tokens without re-involving the user.
Authorization Code Flow + PKCE — Deep Dive

The most secure flow for user-facing applications. PKCE (Proof Key for Code Exchange, RFC 7636) was originally designed for mobile apps but is now required for ALL clients including web apps, to defend against authorization code interception attacks.

PKCE Step-by-Step
1
Generate code_verifier
Random string, 43-128 chars, URL-safe
e.g. dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk
2
Derive code_challenge
code_challenge = BASE64URL(SHA256(code_verifier))
E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM
3
Send challenge in auth request
Auth server stores the challenge against the session
4
Send verifier in token exchange
Auth server hashes verifier → must match stored challenge
If stolen auth code used without verifier → rejected
Client Credentials Flow

Used when there is no user involved — your backend microservice needs to call another service's API. The client authenticates directly with its own credentials.

POST https://auth.example.com/token grant_type=client_credentials client_id=service_a_id client_secret=service_a_secret scope=read:reports write:jobs // Response: access_token (no refresh_token)
Device Authorization Flow

The device cannot open a browser (e.g. Apple TV, CLI tool). It gets a user_code to display, then polls for the token while the user authorizes on another device.

// Step 1: Device requests code POST /device/code client_id=tv_app &scope=streaming // Response: { "device_code": "GmRh...", "user_code": "WDJB-MJHT", ← shown on TV "verification_uri": "example.com/device" } // Step 2: Device polls every 5s until approved POST /token grant_type=urn:ietf:params:oauth:grant-type:device_code
Refresh Token Grant

Access tokens expire (typically in 1 hour). Rather than sending the user through the whole login flow again, you silently exchange the refresh token for a new access token.

POST https://auth.example.com/token grant_type=refresh_token refresh_token=1//0eXm8... client_id=your_app client_secret=your_secret // Returns new access_token (and maybe new refresh_token)
Three Types of Tokens

OAuth 2.0 uses different tokens for different purposes. Understanding their differences is critical for both security and implementation.

Access Token short-lived · ~1hr
The credential used to access protected resources. Sent in every API request. Should be short-lived to limit damage if leaked.
Format JWT or opaque string
Sent via Authorization: Bearer <token>
Revoke? Hard (JWTs are stateless)
Refresh Token long-lived · days/weeks
Used only to get new access tokens. Never sent to resource servers. Store securely — if leaked, attacker can get fresh access tokens.
Format Opaque string
Sent to Auth server only
Revoke? Yes — call /revoke endpoint
ID Token (OIDC) always JWT
Part of OpenID Connect. Contains identity claims (who the user is). Never sent to APIs — only read by your app to establish the user's identity.
Format JWT (always)
Contains sub, email, name, picture...
Use for Authentication (not API calls)

⚠️ JWT Signature Verification is Critical

Anyone can decode a JWT (it's just base64). The signature is what proves authenticity. Your resource server must always verify the signature using the auth server's public key. Never trust a JWT without verifying it.

🔒
Always Use PKCE
Even for confidential clients (server-side apps). PKCE prevents authorization code interception attacks where an attacker intercepts the code in the redirect.
🔄
Validate the state Parameter
Generate a random opaque value, store it in session, and verify it matches on callback. This prevents CSRF attacks where an attacker initiates the flow.
📍
Whitelist redirect_uri
Register exact redirect URIs with the auth server. Never use open redirectors. A misconfigured redirect can let attackers steal auth codes.
🎯
Use Minimal Scopes
Only request the scopes your app actually needs. Principle of least privilege — if you only need email, don't request calendar access.
⏱️
Short Access Token Lifetimes
Keep access tokens short-lived (15-60 min). A leaked access token has limited usefulness if it expires quickly. Use refresh tokens for longer sessions.
🔐
Secure Token Storage
In browsers: use HttpOnly cookies for tokens (not localStorage). In mobile apps: use the platform's secure keychain. Never store tokens in plaintext files.
🚫
Deprecated: Implicit Flow
The implicit flow (response_type=token) returns tokens directly in the URL fragment — which is logged and visible to the browser history. Use Auth Code + PKCE instead.
🚫
Deprecated: Password Grant
The Resource Owner Password Credentials grant requires the user to give their password to the app — defeating the entire purpose of OAuth. Never use this in new systems.
⚠️
Token Leakage via Referrer
Tokens in URLs (query params) can leak via Referrer headers. Always use POST bodies or headers to transmit tokens, never URL query parameters.
The mix-up Attack — A Subtle Threat

If an app supports multiple auth servers, an attacker can trick it into sending a code obtained from Server A to Server B. The fix: always bind the authorization request to a specific issuer and validate the iss parameter in the response (RFC 9207).
