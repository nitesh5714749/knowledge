00 — OVERVIEW
What is DynamoDB?
A fully managed NoSQL database by AWS. Fast, scalable, and schemaless — built for applications that need consistent performance at any scale.
📦
Section 01
Tables, Items & Attributes
The building blocks. Tables hold items (rows), items hold attributes (fields). The key difference: every item can have different attributes — no rigid schema.
🔑
Section 02
Primary Keys
Every item needs a primary key. Choose between a simple partition key or a composite partition + sort key. The hash function determines physical storage placement.
🔍
Section 03
Secondary Indexes
Query your data by attributes other than the primary key. GSIs span the whole table with different keys. LSIs keep the same partition key but different sort key.
⚡
Section 04
DynamoDB Streams
An ordered log of every data change — insert, modify, delete — in near real-time. Pair with Lambda to trigger reactions to data events.
🔬
Section 05
Consistency Internals
Why LSI supports strong consistency but GSI does not. Rooted in physical co-location — same partition node vs async projection to a separate one.
🌍
Section 06
Global Tables
The same table replicated across multiple AWS regions. Read and write locally for low latency. Choose eventual or strong cross-region consistency.

01 — TABLES, ITEMS & ATTRIBUTES
The core building blocks
DynamoDB stores data in tables. Each table holds items, and each item holds attributes. Think of it like a database table — but flexible per row.
Container
Table
A collection of data — like a table in SQL. Examples: People, Music, Orders. You define the primary key, the rest is flexible.
Row equivalent
Item
A single record in a table. Each item is uniquely identified by its primary key. No limit to the number of items. Max item size is 400 KB.
Column equivalent
Attribute
A key-value pair within an item. Can be scalar (String, Number, Binary) or complex (List, Map, Set). Nesting up to 32 levels deep.

02 — PRIMARY KEYS

      
Uniquely identifying items

      
The primary key uniquely identifies each item. DynamoDB hashes it to decide which physical partition stores the data.

    


    

      

        
Type A — Simple

        
Partition Key only

        
One attribute uniquely identifies each item. No two items can share the same partition key value.

        

          Partition Key
        

        

          Example: People table

          PK = PersonID (101, 102, 103...)
        

      

      

        
Type B — Composite

        
Partition Key + Sort Key

        
Two attributes combined. Multiple items can share the same partition key — but must differ on sort key. Unlocks powerful range queries.

        

          Partition Key
          Sort Key
        

        

          Example: Music table

          PK = Artist + SongTitle
        

      

    


    

      

        

          

          Simple PK — People table
        

        
[
  { "PersonID": 101, "Name": "Fred" },   // PersonID 101 → unique
  { "PersonID": 102, "Name": "Mary" },  // PersonID 102 → unique
  { "PersonID": 103, "Name": "Howard" } // PersonID 103 → unique
]

        
✦ PersonID is hashed → determines which partition stores this item

      

    


    

      

        

          

          Composite PK — Music table
        

        
[
  { "Artist": "The Beatles", "SongTitle": "Hey Jude" },
  { "Artist": "The Beatles", "SongTitle": "Let It Be" },
  // ↑ Same Artist (partition key) — OK because SongTitle (sort key) differs
  { "Artist": "Radiohead",   "SongTitle": "Creep" }
]

        
✦ Query: Artist = "The Beatles" → returns ALL their songs sorted by SongTitle

03 — SECONDARY INDEXES

      
Query beyond the primary key

      
Secondary indexes give you alternative query paths. You design your table around primary access patterns, then add indexes for secondary ones.

    


    

      

        GSI — Global Secondary Index
        
Completely different keys

        
A GSI can have a totally different partition key and sort key from the base table. It spans the entire table — all partitions.

        
    
          
  • Partition keyany attribute
    •

        LSI — Local Secondary Index
        
Same partition, new sort

        
An LSI keeps the same partition key as the base table but uses a different sort key. "Local" means it only queries within one partition.

Important concept

      
Projection

      
When creating an index, you choose which attributes to project (copy) into it. At minimum, the primary key attributes are always projected. You can also project ALL attributes or a specific INCLUDE list — more projection means faster reads but more storage cost.

    

  


  
  

    

      
04 — DYNAMODB STREAMS

      
React to data changes

      
Streams captures every modification to your table as an ordered, time-stamped log. Optional feature — enable per table. Records expire after 24 hours.

    


    

      

        
Retention

        
24h

        
Stream records automatically expire after 24 hours.

      

      

        
Ordering

        
Per-item ordered

        
Changes to the same item appear in strict order. Cross-item ordering is not guaranteed.

      

      

        
Latency

        
Near real-time

        
Events appear in the stream within milliseconds of the write completing.

05 — CONSISTENCY INTERNALS

      
Why LSI can be strongly consistent but GSI cannot

      
This is an architectural decision rooted in where data physically lives. Click each index type below to understand the internal mechanics.

    


    
    

      
      
      
    


    

      

        LSI — Local Secondary Index
        
Same physical partition = sync possible

        
An LSI shares the same partition key as the base table. This means the LSI data lives on the same physical partition node as the base table data.

        

          

            
BASE TABLE PARTITION

            
Item writes land here. The LSI index entry sits on the same node. Both can be updated in a single atomic operation.

          

          

            
LSI INDEX (same node)

            
Because it's co-located, a write commits both the base item and the LSI entry together — no lag. Strong consistency is possible.

✦ Strong consistency supported: Table ✓  |  LSI ✓  |  GSI ✗  |  Streams ✗

    


    

      

        GSI — Global Secondary Index
        
Different partition key = physically separated

        
A GSI can have a completely different partition key from the base table. This means GSI data is distributed across different physical partition nodes — nodes the base table write doesn't control.

        

          

            
BASE TABLE PARTITION

            
Write commits here immediately. Base table is up to date instantly.

          

          

            
GSI PARTITION (different node)

            
Update is projected asynchronously. A small lag exists before the GSI reflects the latest state — hence no strong consistency guarantee.

✦ GSI reads are always eventually consistent — there is no API option to request strong consistency on a GSI.

Streams — asynchronous by design

          
DynamoDB Streams is an event log that captures changes after they are committed to the base table. The stream record is written independently, after the write completes — making strong consistency reads on a stream impossible by design.

06 — GLOBAL TABLES

      
One table, multiple regions

      
A Global Table is the same DynamoDB table deployed across multiple AWS regions simultaneously. Each region hosts a full replica — readable and writable.

    


    

      

        🌍
        
Multi-region

        
Same table, every region

        
Same name, same primary key schema deployed in us-east-1, eu-west-1, ap-southeast-1, etc. Users always hit their nearest region.

      

      

        ⚡
        
Low latency

        
Read & write locally

        
A user in Tokyo reads and writes to the Tokyo replica — single-digit millisecond latency, not a round-trip to us-east-1.

      

      

        🛡️
        
High availability

        
Region failure? No downtime

        
If one region goes offline, all other replicas continue serving traffic. Your app routes around the failure automatically.

      

      

        🔄
        
Auto-replication

        
Writes propagate everywhere

        
A write in Frankfurt is automatically replicated to all other regions. You don't manage this — DynamoDB handles it.

Design rules & trade-offs

    

    

      

        
Hard constraint

        
Schema must match

        
All replicas share the exact same primary key and sort key schema. You cannot have different key structures per region.

      

      

        
Trade-off

        
Conflict handling

        
Two regions can accept writes to the same item simultaneously. DynamoDB uses last-writer-wins (by timestamp) to resolve conflicts. Plan your write patterns carefully.

      

      

        
Cost

        
Replicated WCUs

        
Each write is consumed in every replica region. A write to a 3-region global table costs 3× the WCU of a single-region write.

      

      

        
Throughput

        
Plan per region

        
Each replica needs enough provisioned or on-demand capacity to handle the full workload — replicated writes count against the destination region's capacity too.
