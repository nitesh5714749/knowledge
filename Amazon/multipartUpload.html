Concept

The core idea

Instead of sending a huge file in one big HTTP request (which can time out, fail, or be slow), multipart upload splits the file into independent chunks, uploads them in parallel, then asks the server to stitch them back together.

💡
This is a classic computer science pattern — divide and conquer. Break a large problem into smaller, independently solvable sub-problems. Uploading a 2 GB file as 20 × 100 MB chunks lets you exploit parallelism and retry only what fails.

Process

The 3-step process

Every multipart upload — whether AWS S3, GCS, or custom — follows this exact three-phase lifecycle.

1
🔑
Initiation
The client tells the server: "I'm about to upload a large file." The server replies with a unique Upload ID — a token that links all future chunk requests together. Think of it as opening a tab at a restaurant.
POST /uploads
← { uploadId: "abc-123" }
2
📤
Chunk uploads
The client splits the file and uploads each part independently — possibly on multiple threads at once. Each request carries the Upload ID, a part number, the chunk data, and a checksum. Parts can arrive out of order. If one fails, only that part is retried.
PUT /uploads/abc-123/parts/3
← { ETag: "xyz789", checksum: "..." }
3
🔧
Completion
Once all parts are confirmed, the client sends a complete request listing all part numbers and their ETags. The server assembles them in order, verifies the final checksum, and the object is available for access — just like any normally uploaded file.
POST /uploads/abc-123/complete
{ parts: [1,2,...,N], etags: [...] }

Benefits

Why it works so well

Four properties that make multipart uploads the right choice for large files.

⚡
High throughput
By uploading N parts in parallel, you fully saturate your available bandwidth instead of being limited by a single TCP stream. A 5-worker upload can be ~5× faster than sequential.
🔄
Fault tolerance
If part 7 of 20 fails, only part 7 is retried. You don't restart from byte 0. On spotty networks (mobile, satellite), this is a massive practical win.
🛡️
Data integrity
Each part ships with a checksum (MD5 or SHA-256). The server verifies every single part before assembly. If bytes got corrupted in transit, the mismatch is caught immediately.
⏸️
Pause & resume
Since parts are independent, you can stop an upload and resume it hours later. The server keeps confirmed parts indefinitely until you complete or abort the upload.

AWS S3

S3-specific details

Amazon S3's multipart upload has a few important rules you need to know in practice.

🏷️
Best practice threshold: 100 MB. AWS recommends using multipart upload for objects 100 MB or larger. Below that, a single PUT request is simpler and fast enough.
⏳
No auto-expiry on incomplete uploads
Once initiated, an S3 multipart upload has no expiry. You must explicitly complete or abort it — or you'll pay for the stored parts forever. Use an S3 Lifecycle rule to auto-abort stale uploads after N days.
🔐
Automatic checksums
Updated AWS clients automatically calculate and send a checksum for every part. S3 verifies each part on arrival, catching any in-transit corruption before assembly begins.
🔀
Any order, any time
Parts can be uploaded in any order. Part 7 can arrive before part 1. S3 tracks which parts have landed and only assembles when you send the complete request with the final ordered list.
🏗️
Upload before knowing final size
You can begin uploading parts while still generating the object — perfect for live video transcoding or log aggregation where the final size isn't known until the stream ends.
