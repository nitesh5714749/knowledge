01 — Foundations

What even is a socket?

A socket is just a file descriptor — a handle the OS gives your process to represent one end of a network connection. When your browser connects to a server, both sides get a socket. Data written into one end comes out the other.

Under the hood, it's a TCP connection — a reliable, ordered, bidirectional byte stream between two IP:port pairs. The OS manages the actual sending and receiving via kernel buffers. Your application just reads and writes.

Key insight
An open socket is not constant work. It's state. It consumes a file descriptor, some kernel memory for buffers, and TLS state if encrypted — but zero CPU until data actually arrives. This is the most important thing most engineers miss.

02 — TCP Identity

The 4-tuple

Here's the thing that confuses people: the server always listens on the same port (say :8080). So how does it tell 10,000 connections apart? The answer is — the OS doesn't identify a connection by port alone. It uses all four values together.

A TCP connection is uniquely identified by
Client IP
192.168.1.5
Client port — changes every request
:54231
Server IP
10.0.0.1
Server port — always same
:8080

As long as any one of these four values differs, it's a completely different connection in the OS's eyes. The server port never changes, but the client port does — and that's enough.

Three requests from one client — three different connections
192.168.1.5:54231 → 10.0.0.1:8080 GET /users
192.168.1.5:54232 → 10.0.0.1:8080 POST /order
192.168.1.5:54233 → 10.0.0.1:8080 GET /status
03 — Port Lifecycle

Ephemeral ports and TIME_WAIT

The OS picks those changing client ports from the ephemeral port range — on Linux this is typically 32768–60999. That's about 28,000 ports. Once you exhaust them, new connections fail. This is called port exhaustion.

# check the range on your machine
cat /proc/sys/net/ipv4/ip_local_port_range
→ 32768   60999

# see current TCP connections
ss -tan state established

# check open file descriptors
ulimit -n
TIME_WAIT — the silent killer
After a TCP connection closes, the port enters TIME_WAIT state for ~60 seconds before being released back to the pool. If your backend service makes thousands of short-lived outbound connections per second (e.g. hitting a database without connection pooling), you will run out of ports. This is a very real production incident.

The fix is almost always connection pooling — reuse existing connections instead of opening new ones per request. This is why every database driver, HTTP client library, and Redis client has a pool setting.

04 — HTTP Keep-Alive

Keep-alive: reusing the same connection

Without keep-alive (HTTP/1.0 style), every request pays the full TCP handshake cost: SYN → SYN-ACK → ACK. That's at least one full round-trip before your actual data moves. On a 50ms connection, that's 50ms of pure overhead per request.

Without keep-alive (HTTP/1.0)
SYN → SYN-ACK → ACK → request → response → FIN. New ephemeral port every time. 3 round-trips minimum per request just for setup.
With keep-alive (HTTP/1.1 default)
Same TCP connection reused for multiple requests. Port stays constant. Handshake paid once. HTTP/1.1 enables this by default with Connection: keep-alive.
WebSocket — the permanent upgrade
One HTTP connection upgraded to WebSocket via 101 Switching Protocols. After that, both sides can push data freely without request/response cycles. One port, forever open.
05 — Redis Internals

Redis: single-threaded and blazing fast

Redis uses one main thread for all command processing. Not two. Not a pool. One. And yet it handles hundreds of thousands of operations per second. How?

The trick: each command finishes in ~1–10 microseconds (pure RAM access). By the time you'd even context-switch between threads, Redis has already served 10 commands on its single thread. The overhead of locking, mutex contention, and context switches would cost more than the work itself.

The real win: no locks needed
Because only one command runs at a time, there's no "two threads writing the same key" scenario. No mutexes. No deadlocks. No race conditions — by design. INCR counter called by 1000 clients will always produce exactly the right answer.

How 100 clients get served by 1 thread

Redis doesn't give each client a thread. Instead it uses epoll to watch all client sockets simultaneously. When a socket has data ready, the OS wakes the thread. The thread processes that command, replies, then checks for the next ready socket.

Ready queue at any given moment
SET name "Alice" GET counter INCR counter SET key2 "Bob" EXPIRE k 30 GET user:99

The thread works through this left to right, one at a time. No parallelism needed — each item takes microseconds.

# These two commands from different clients are ALWAYS safe
# because only one runs at a time

Client A: INCR counter   → counter was 5, now 6
Client B: INCR counter   → counter was 6, now 7

# Never: both read 5, both write 6 (that's a multithreaded bug)
06 — OS Machinery

epoll: the OS watching all your sockets

Both Redis and WebSocket servers rely on the same OS primitive: epoll on Linux, kqueue on macOS/BSD. This is the thing that makes single-threaded servers handling thousands of connections possible.

The pattern is simple. Instead of assigning a thread per connection (which would mean 100K threads for 100K connections — a disaster), you register all your sockets with epoll and ask a single question:

The epoll question
"Which of my registered sockets are ready to read or write right now?"

The OS returns only the sockets that have data. Your thread does work for those sockets and goes back to waiting. This is called non-blocking I/O or the reactor pattern.
Thread registers 100K sockets with epoll
Zero CPU cost. The kernel keeps a watch list.
Client sends data → kernel marks socket "readable"
Hardware interrupt → kernel wakes epoll → returns the ready socket to your thread. No polling. No spinning.
Thread wakes, processes only that socket
Reads the command or message, does the work, sends the reply. Then sleeps again. Other 99,999 sockets were undisturbed.
// Node.js heartbeat — detecting dead WebSocket connections
import { WebSocketServer } from 'ws'

const wss = new WebSocketServer({ port: 8080 })

wss.on('connection', (ws) => {
  ws.isAlive = true
  ws.on('pong', () => { ws.isAlive = true })
})

setInterval(() => {
  for (const ws of wss.clients) {
    if (!ws.isAlive) { ws.terminate(); continue }
    ws.isAlive = false
    ws.ping()   // if no pong in 30s → dead
  }
}, 30000)
07 — Scale

100K WebSocket connections

Yes — a system like Discord really does maintain ~100K (or more) live WebSocket connections at once. And yes, one user on a laptop, phone, and browser tab counts as 3 connections. A reconnect creates a new session. A dead mobile connection may linger as a ghost session until heartbeat timeout.

The crucial distinction
Connection count ≠ message rate. 100K open sockets is a memory problem (~6GB). 100K messages/second is a CPU + fanout problem. These are completely different dimensions of scale. Confusing them makes the architecture seem impossible when it's actually two separate, solvable problems.

Production fleet layout

No one puts 100K connections on a single machine. The standard pattern is a fleet of gateway servers behind a load balancer. Each gateway owns a slice of connections. A pub-sub bus (Redis, Kafka, NATS) routes events between gateways when a message needs to reach users on a different machine.

Separation of concerns in a real-time system
Gateway layer — connection ownership
Hold sockets, track sessions, send heartbeats, detect dead peers. No business logic. Just connections.
App layer — business logic
Validate messages, persist to DB, determine recipients. Stateless, horizontally scalable.
Routing layer — event distribution
Pub-sub bus (Redis pub/sub, Kafka, NATS) that tells the right gateway to push the event to the right sessions.

Fanout: the actual hard problem

Holding sockets is easy. The real engineering challenge is fanout — when one event needs to reach 50,000 users. The system must validate the send, persist the message, look up which gateways own those sessions, enqueue work to each gateway, and write to thousands of sockets — all without letting a few slow clients block everyone else.

Presence is nastier than messaging
Who is online right now? On which device? In which room? Which gateway owns that session? Did the user disconnect, or did their train enter a tunnel and leave a dead TCP session that won't expire for 60 seconds? Presence is distributed state management wearing a simple UI feature as a disguise.
08 — Production Reality

What breaks first

In practice, you hit these problems in roughly this order — and none of them are "the server ran out of sockets":

1. File descriptor limit (ulimit -n)
Default is 1024 on most Linux installs. Hit this before you hit any real architectural problem. Fix: ulimit -n 1000000 or set in /etc/security/limits.conf.
2. Memory from TLS + buffers
~60KB × 100K connections = ~6GB RAM. Fine on modern hardware, not on a cheap VM. TLS context and compression state are the biggest per-connection cost.
3. Slow consumers and backpressure
A few clients that can't drain writes fast enough cause write buffers to fill. Without proper backpressure, the server becomes a queue by accident.
4. Hot fanout
One celebrity, one very large room, one broadcast event — traffic shape the architecture didn't plan for. CPU and bandwidth spike simultaneously.
5. Reconnect storms — the worst one
A deploy, network blip, or TLS incident forces 30% of clients to reconnect in 90 seconds. Steady-state connection count was fine. Recovery path wasn't.
The question to ask
Don't just ask "how many sockets can this box hold?" Also ask: "What happens when 30% of them reconnect in 90 seconds?" — that's usually where real systems fall over.
