01 — Fundamentals

What is a Webhook?

A webhook is a reverse API — instead of you polling for data, the server pushes data to you the moment something happens.

❌ Regular API (polling)
# Your server asks every 30 seconds:
GET /payment/status?id=pay_123
# → "not yet"
GET /payment/status?id=pay_123
# → "not yet"
GET /payment/status?id=pay_123
# → "paid!" (finally, 90s later)
✓ Webhook (push)
# Razorpay calls YOUR server:
POST https://yourstore.com/webhook
{
  "event": "payment.success",
  "payment_id": "pay_123",
  "amount": 50000
}
# → instant! no polling needed
The analogy
Polling = calling the restaurant every 5 mins asking "is my food ready?"

Webhook = the restaurant calls you when your order is ready.

03 — Problems Identified

What Broke at Scale

As transactions grew exponentially, five distinct failure modes emerged. Each one needed a targeted fix.

💥
DB Connection Bottleneck
MySQL has a hard limit on concurrent connections (~300). 500 workers all writing simultaneously = 200 workers blocked, queuing, and spiking latency from 2s → 4s.
📈
Traffic Spikes
IPL, Big Billion Day, Quarter Close — unpredictable bursts that overwhelm a system designed for average load. Auto-scaling workers made DB worse.
🗄️
Read Degradation
100M+ rows in the webhook table. Queries slow even with indexes. Data no longer fits in RAM → reads hit disk. Replica also lags under high write pressure.
🚦
No Priority
One SQS queue for everything. A burst of low-priority SMS events blocks critical payment webhooks. No way to say "this is urgent".
⏳
Slow Customer Servers
Workers block synchronously waiting for merchant HTTP responses. One slow merchant with 8-minute response times ties up N workers, starving everyone else.

07 — Solution 3

Quality of Service (QoS)

Each merchant gets a QoS score based on their server's response time. Slow merchants are auto-demoted. Fast merchants are auto-promoted. Watch it live.

Merchant QoS score Score Queue
→ Click "Simulate webhook round" to see QoS in action
Demotion rule
Response time > 5 min → score drops. Score ≤ 3 → moved to P2. Events still delivered, just slower.
Promotion rule
Probe sent every N mins. Fast response → score rises. Score ≥ 8 → promoted back to P0. Fully automatic.

Why this connects back to everything we discussed
Slow merchant server
When Swiggy's webhook endpoint is slow to respond with 200 OK, Razorpay's workers block waiting. That's the QoS problem.
Rogue burst
During Big Billion Day, Flipkart fires 500k webhooks at once — flooding the queue. That's the rate limiter problem.
Retry logic
If Swiggy's server is down and returns 500, Razorpay retries the webhook so Swiggy never misses a payment event.
SLA matters
If Razorpay can't deliver webhooks fast, Swiggy doesn't know you paid → your order never gets placed. That's why SLAs are non-negotiable.
The restaurant analogy
Without webhook (polling): Swiggy calls Razorpay every 5 seconds — "did the customer pay yet? did they pay yet? did they pay yet?"

With webhook: Razorpay calls Swiggy the moment payment is confirmed — "payment done, here are the details." One call, instant.
