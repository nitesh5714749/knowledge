Kernel Buffers

Every TCP socket gets its own send buffer and recv buffer allocated in the kernel at handshake time.

SEND BUFFER
send buf ~16KB defaultkernel owned
REQ bytes
free...
  • ✅ App writes bytes → kernel queues here
  • ✅ TCP drains automatically → sends to wire
  • ✅ Waits for ACK per segment sent
  • ✅ You never touch this directly
  • → drains on its own, no action needed
RECV BUFFER
recv buf ~87KB defaultapp must drain!
RESP bytes arriving
free
  • ✅ Bytes arrive from wire → kernel places here
  • ❌ Does NOT drain automatically
  • ❌ App MUST read to empty it
  • ⚠️ If full → TCP sends WINDOW=0
  • ⚠️ WINDOW=0 → server pauses → deadlock
LAYER STACK — what sits above the buffers
┌─────────────────────────────────────────────────────────┐
│  APP LAYER  — your .NET code                            │
│  HttpClient.SendAsync()  /  ReadAsStringAsync()         │
└────────────────────────┬────────────────────────────────┘
                         │ reads/writes
┌────────────────────────▼────────────────────────────────┐
│  HTTP LAYER — formats requests, parses responses        │
│  adds headers, reads Content-Length, handles chunked    │
└────────────────────────┬────────────────────────────────┘
                         │ byte stream
┌────────────────────────▼────────────────────────────────┐
│  SEND BUFFER (kernel)   ~16KB                           │
│  [ bytes waiting to go to wire         | free      ]    │
│  TCP drains → sends segments → waits for ACKs           │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  RECV BUFFER (kernel)   ~87KB                           │
│  [ bytes arrived from wire             | free      ]    │
│  Sits here until YOUR APP reads it                      │
└────────────────────────┬────────────────────────────────┘
                         │ app reads
┌────────────────────────▼────────────────────────────────┐
│  HTTP LAYER — parses response bytes                     │
└────────────────────────┬────────────────────────────────┘
                         │
┌────────────────────────▼────────────────────────────────┐
│  APP LAYER — you get HttpResponseMessage                │
└─────────────────────────────────────────────────────────┘
The golden asymmetry: Send buffer drains itself. Recv buffer needs you. This asymmetry is the root cause of every CLOSE_WAIT and connection reuse problem in .NET.

Sockets & Ports

Buffers are created per socket, not per port. One port can have thousands of sockets — each with its own independent buffers.

SOCKET IDENTITY — the 4-tuple
Socket = unique 4-tuple

  ( src IP  :  src port  :  dst IP  :  dst port )
  ─────────────────────────────────────────────
  192.168.1.10 : 54321  : 93.184.16.34 : 443   ← socket A
  192.168.1.10 : 54322  : 93.184.16.34 : 443   ← socket B  (different!)
  192.168.1.10 : 54321  : 93.184.16.34 : 80    ← socket C  (different!)

Change ANY ONE value = completely different socket = separate buffers
ONE PORT → MANY SOCKETS → MANY BUFFERS
SERVER KERNEL — port 443

  LISTEN socket (port 443)
        │
        ├── Socket fd=7  client-A:54321  send buf A + recv buf A
        │
        ├── Socket fd=8  client-B:54322  send buf B + recv buf B
        │
        └── Socket fd=9  client-C:54323  send buf C + recv buf C

All sharing port 443. Completely isolated buffers.
Filling recv buf A does NOT affect recv buf B at all.
BUFFER LIFECYCLE
SYN-ACK sent
    ↓
Kernel allocates socket struct
Kernel allocates send buf (~16KB)
Kernel allocates recv buf (~87KB)
    ↓
ESTABLISHED — buffers ready
    ↓
Data flows...
    ↓
Socket → CLOSED
Kernel FREES both buffers ✅

CLOSE_WAIT stuck?
Buffers NEVER freed ❌
Memory leak forever
DEFAULT SIZES (Linux)
/proc/sys/net/ipv4/tcp_rmem
recv buf:  4KB / 87KB / 6MB
            min  def   max

/proc/sys/net/ipv4/tcp_wmem
send buf:  4KB / 16KB / 4MB
            min  def   max

Per connection cost: ~100KB
1000 conns  = ~100MB
10000 conns = ~1GB ← CLOSE_WAIT cost

Kernel auto-tunes in range

Client closes first: Client gets TIME_WAIT (harmless). Server gets CLOSE_WAIT. This happens when your HttpClient's PooledConnectionIdleTimeout expires before server's keepalive timeout. 
Q: Does FIN go into the recv buffer alongside body data?

A: NO. FIN is a TCP control flag, not data bytes.

FIN packet structure:
  IP header
  TCP header → FIN flag = 1, len = 0  ← zero data bytes
  no payload

Recv buffer before FIN arrives:
┌────────────────────────────────────────────────┐
│ [body byte 0..99999]                           │
│  read_ptr=0  write_ptr=100000                  │
└────────────────────────────────────────────────┘

Recv buffer AFTER FIN arrives:
┌────────────────────────────────────────────────┐
│ [body byte 0..99999] [EOF]                    │
│  UNCHANGED — same bytes + virtual EOF marker   │
└────────────────────────────────────────────────┘

EOF = not real bytes. Virtual marker placed by kernel.
Kernel also IMMEDIATELY sends ACK — no app needed.
TCP state moves to CLOSE_WAIT automatically.
App has NO idea. Still blocked on await.
FIN processing is 100% kernel. Kernel receives FIN → places EOF marker in recv buffer → sends ACK immediately → changes TCP state to CLOSE_WAIT. Your app wakes up only when it reads to the EOF marker.

FIN_WAIT_1 & FIN_WAIT_2

These states only appear on the side that closes FIRST. They are the normal, expected sequence for the closer.

FIN_WAIT_1
FIN_WAIT_2
Can they get stuck?
Server closes first
FIN_WAIT_1

How you enter:  You called close() / Dispose() → kernel sent FIN
What it means:  "I sent goodbye — waiting for their ACK"
Can receive:    YES — connection is half-closed, other side can still send
Waiting for:    ACK of your FIN → then moves to FIN_WAIT_2
Duration:       Usually milliseconds

Stuck scenario: Remote crashed before sending ACK
                TCP retransmits FIN automatically
                Eventually times out → CLOSED

netstat shows:
  tcp  client:54321  server:443  FIN_WAIT1  ← client sent FIN, waiting ACK
FIN_WAIT_2

How you enter:  Were in FIN_WAIT_1, remote sent ACK for your FIN
What it means:  "They acknowledged my goodbye — waiting for THEIR goodbye"
Can receive:    YES — still half open, remote can send data
Waiting for:    Remote's FIN → send ACK → move to TIME_WAIT
Duration:       Depends on remote application

Stuck scenario:  Remote sent ACK but never calls close()
                On Linux: tcp_fin_timeout = 60s → kernel forces close
                On Windows: can stay MUCH longer ⚠️

netstat shows:
  tcp  client:54321  server:443  FIN_WAIT2
  tcp  server:443  client:54321  CLOSE_WAIT  ← server stuck meanwhile
FIN_WAIT_1 stuck?  Rare — means remote not ACKing
  Cause:    network issue / remote crashed
  Fix:      TCP retransmits automatically, eventually times out

FIN_WAIT_2 stuck?  Possible — remote never calls close()
  Cause:    remote app bug — got ACK but never disposes
  Fix:      Linux auto-closes after tcp_fin_timeout (60s)
            Windows: configure tcp_fin_timeout manually

CLOSE_WAIT stuck?  Most dangerous — YOUR app never calls close()
  Cause:    missing using/Dispose in YOUR code
  Fix:      fix your code — NEVER auto-closes on its own

Rule: FIN_WAIT_1/2 are transient (milliseconds to seconds)
      CLOSE_WAIT is permanent until app calls Dispose()
When server closes first — WHO gets WHAT state:

            CLIENT          SERVER
            ──────          ──────
Step 1:     ESTABLISHED     FIN_WAIT_1   ← server sent FIN
Step 2:     CLOSE_WAIT      FIN_WAIT_2   ← client ACKed
Step 3:     LAST_ACK        TIME_WAIT    ← client sent FIN
Step 4:     CLOSED          TIME_WAIT    ← 60s then CLOSED

Server gets:  FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT
Client gets:  CLOSE_WAIT → LAST_ACK → CLOSED

TIME_WAIT (server): harmless — 60s then gone ✅
CLOSE_WAIT (client): dangerous — stuck if no Dispose() ❌

In .NET with Kestrel:
  Kestrel keepalive = 120s
  Your client idle timeout should be < 120s (e.g. 90s)
  Client closes first → Client gets TIME_WAIT (harmless)
  Server gets CLOSE_WAIT briefly then closes cleanly ✅
// the danger state

CLOSE_WAIT

The zombie state. Not idle, not active. Consuming resources and doing nothing. Only your code can fix it.

Is it IDLE?
Idle = ready to work
       just waiting

CLOSE_WAIT:
NOT ready to work
Cannot be reused
Not truly idle
Is it ACTIVE?
Active = data flowing
         doing work

CLOSE_WAIT:
No data flowing
Nothing useful
Not active
It is a ZOMBIE
Looks alive (socket exists)
Has a port (fd allocated)
Cannot send (server gone)
Cannot recv (server closed)
Not in pool (unusable)
Wasting port+mem+fd
WHAT EACH CLOSE_WAIT CONSUMES
FILE DESCRIPTOR
  OS limit ~65535 per process
  Each CLOSE_WAIT uses 1
  Too many → "too many open files" 💥

PORT NUMBER
  Range ~28000-65535
  Each socket holds 1 port
  Too many → port exhaustion
  New connections fail 💥
MEMORY
  ~8KB send buf + ~87KB recv buf
  100 sockets  = ~9MB wasted
  10000 sockets = ~900MB 💥

KERNEL TRACKING
  OS tracks every socket
  More sockets = kernel overhead
  Slows ALL network operations 💥
CLOSE_WAIT count rule: 0–10 = transient, acceptable. >20 growing = warning. >100 or continuously growing = critical bug in your code. netstat -an | findstr CLOSE_WAIT
// http/1.1 keep-alive

Keep-Alive & Buffer Drain

Keep-alive reuses the same socket. Same socket = same recv buffer. The buffer must be empty before the next response arrives.

WHY DRAIN IS MANDATORY
Keep-alive  →  same socket reused for all requests
Same socket →  same recv buffer
Same buffer →  ONE continuous byte stream

Request 1 response → goes into recv buffer
Request 2 response → goes into SAME recv buffer

They CANNOT overlap. Buffer must be empty between requests.

If Resp1 body not drained before Req2 response arrives:

  recv buffer:
  ┌────────────────────────────┬──────────────────┐
  │ RESP1 body (stuck 100KB)   │ RESP2 bytes ???  │
  └────────────────────────────┴──────────────────┘

  HTTP parser reads Resp1 body bytes thinking they are Resp2
  CORRUPT DATA 💥
  OR buffer fills → WINDOW=0 → server pauses → DEADLOCK 💥
RING BUFFER MECHANICS
recv buffer is a ring buffer in kernel:

  write_ptr = where kernel writes new arriving bytes
  read_ptr  = where app reads from
  free space = total - (write_ptr - read_ptr)

HEALTHY — body drained:
  ┌──────────────────────────────────────────┐
  │  [                  free               ] │
  │   ↑ read_ptr = write_ptr                 │
  └──────────────────────────────────────────┘
  Req2 response fills from beginning ✅

BROKEN — body NOT drained:
  ┌──────────────────────────────────────────┐
  │  [resp1 body 100KB stuck   |  free     ] │
  │   ↑ read_ptr never moved   ↑ write_ptr   │
  └──────────────────────────────────────────┘
  Req2 bytes pile in after Resp1 body — CORRUPT ❌
TCP WINDOW = BACKPRESSURE
Normal:
  recv buf free = 87KB
  TCP tells server: WINDOW = 87KB  "send up to 87KB"
  Server sends data...

Buffer fills up:
  recv buf free = 0
  TCP tells server: WINDOW = 0  "STOP! no room"
  Server PAUSES ⏸

App reads 50KB:
  recv buf free = 50KB
  TCP tells server: WINDOW = 50KB  "resume, 50KB free"
  Server resumes ▶

If app never reads → WINDOW stays 0 → server frozen
You try to send Req2 → server is frozen waiting for window
DEADLOCK — neither side makes progress 💀

ResponseHeadersRead

The completion option that changes when .NET returns to you — and why it makes the drain problem visible.

✅ ResponseContentRead (default)
// default behavior
var r = await client.GetAsync(url);
// .NET waits until ENTIRE body
// is buffered in memory
// THEN returns to you

recv buf: [empty — already drained ✅]

Connection reusable immediately ✅
Small body (<85KB) auto-buffered ✅
"Accidentally safe" even without using
⚠️ ResponseHeadersRead
// streaming mode
var r = await client.GetAsync(url,
  HttpCompletionOption.ResponseHeadersRead);
// returns as soon as HEADERS arrive
// body still in recv buffer

recv buf: [body bytes still arriving ❌]

YOU must drain or dispose ❌
Without using → connection broken ❌
Large body → WINDOW=0 → deadlock ❌
KESTREL BUFFER SIZE EFFECT
Kestrel default MaxResponseBufferSize = 64KB

Response < 64KB:
  Kestrel buffers ENTIRE response before sending
  Client recv buf gets headers + body at once
  ResponseContentRead: body already buffered → safe ✅
  ResponseHeadersRead: body arrives instantly anyway → "accidentally safe" ⚠️
  → This is why your 199 byte test showed reuse without using!

Response > 64KB:
  Kestrel STREAMS in chunks
  Client recv buf fills gradually
  ResponseHeadersRead: body genuinely in flight ❌
  Not disposing → WINDOW=0 → new connection every request ❌

Never rely on accidental safety. Always use `using`.
Why your production bug was hidden: Your responses were small enough to be auto-buffered by Kestrel AND auto-drained by ResponseContentRead. The bug only surfaces with large responses, ResponseHeadersRead, streaming endpoints, or exceptions thrown mid-stream.

Dispose() & FIN

Dispose() does not send FIN directly. It drains the buffer, then tells the kernel, and the kernel sends FIN.

THE EXACT CHAIN
❌ WRONG mental model:
  app reads EOF → FIN sent

✅ CORRECT mental model:
  Dispose() → kernel → FIN sent

Step by step:

  using var response = await client.GetAsync(url,
    HttpCompletionOption.ResponseHeadersRead);
  var body = await response.Content.ReadAsStringAsync();
  // reading body drains recv buf...
  // hits EOF marker (placed by kernel when FIN arrived)
  // ReadAsStringAsync() returns complete string to you
  // recv buf: empty ✅

  // using block ends HERE ↓
  // Dispose() called automatically
         ↓
  HttpConnection.Dispose()
         ↓
  Socket.Shutdown(Send)  // "I'm done sending"
         ↓
  Kernel: tcp_close()
         ↓
  Kernel sends FIN packet  ← NOT the app
         ↓
  TCP state → LAST_ACK → CLOSED ✅
Path A — body read first
ReadAsStringAsync()
  reads 100KB → EOF
  recv buf empty ✅

using ends → Dispose()
  recv buf: already empty
  Dispose() → kernel
  kernel sends FIN immediately ✅
  CLOSE_WAIT: microseconds
Path B — Dispose() without reading
Body NOT read
  recv buf has 100KB

using ends → Dispose()
  Dispose() internally reads
  and DISCARDS 100KB
  recv buf: drained ✅
  kernel sends FIN ✅
  CLOSE_WAIT: brief ✅

Body lost but conn clean
DOES EOF TRIGGER FIN? — NO
// Proof:
var body = await response.Content.ReadAsStringAsync();
// ^ EOF seen here — body complete
// FIN NOT sent yet!

await Task.Delay(30_000); // 30 seconds pass
// still CLOSE_WAIT — still no FIN

response.Dispose(); // NOW FIN is sent
// ^ Dispose() is the trigger — not EOF

EOF is just a signal to your app:
"server has no more data to send"
It does NOT cause any TCP action.
WHO DOES WHAT
ActionAppKernel
Read body bytes✅ YESserves from recv buf
See EOF✅ YESplaced EOF marker
Call Dispose()✅ YES—
Send FIN packet❌ NEVER✅ ONLY kernel
Send ACK❌ NEVER✅ ONLY kernel
Manage seq numbers❌ NEVER✅ ONLY kernel
Drain recv buf✅ ReadAsyncor Dispose() drains

Rules & Cheatsheet

Everything distilled into rules you can remember and apply immediately.

THE 6 RULES
#RuleWhy
1 Send buffer drains automatically
TCP owns it. You never touch it.		 Kernel manages segments, ACKs, retransmits
2 Recv buffer does NOT drain automatically
Your app MUST read it.		 If full → WINDOW=0 → server pauses → deadlock
3 HTTP/1.1 is strictly sequential
One byte stream, no labels.		 Resp1 body + Resp2 bytes = corruption. Must drain between requests.
4 Dispose() drains recv buf for you
using block = guaranteed drain.		 Internally reads and discards remaining bytes before returning conn to pool
5 FIN is never in the recv buffer
Kernel handles FIN, places EOF marker.		 FIN has len=0. Kernel ACKs it immediately. App sees EOF when it reads to end.
6 Dispose() → kernel → FIN sent
EOF does NOT send FIN. Dispose() does.		 Only when app signals "done sending" does kernel send FIN

✅ ALWAYS DO

  • using var response = ...
  • Read body inside using block
  • Client timeout < Kestrel timeout
  • Configure SocketsHttpHandler
  • Use IHttpClientFactory

❌ NEVER DO

  • using var client = new HttpClient()
  • Ignore response body
  • No finally block on response
  • new HttpClient() per request
  • Rely on GC to close sockets

🔧 KESTREL CONFIG

  • KeepAliveTimeout: 120s
  • MaxResponseBufferSize: 64KB
  • Client IdleTimeout: <120s
  • KeepAlivePingDelay: 30s
  • PooledConnectionLifetime: 2m

📊 MONITOR

  • CLOSE_WAIT should be ~0
  • >20 growing = warning
  • >100 = critical bug
  • netstat -an | findstr CLOSE_WAIT
  • IPGlobalProperties in .NET

⚡ HTTP PORTS

  • HTTP = port 80 (plain text)
  • HTTPS = port 443 (TLS)
  • Buffers same on both ports
  • CLOSE_WAIT same on both
  • TLS adds close_notify before FIN

🧠 MENTAL MODELS

  • Socket = 4-tuple not port
  • CLOSE_WAIT = zombie socket
  • FIN = control flag, not data
  • EOF = marker, not FIN trigger
  • Dispose() = "I'm done" signal
