🚦 How GRL Actually Works

One concrete example, step by step — click Next to walk through it

📖 The scenario: Uber has a Trip Service that other services call to get trip details. It can safely handle 1,000 requests/sec. One day, the Notification Service has a bug and starts sending 5,000 requests/sec — a 5× flood. Without GRL, Trip Service crashes.

Let's watch GRL handle it, step by step. 👇
1
📨
The flood begins
Notification Service
The buggy Notification Service starts firing 5,000 requests/second at Trip Service. Every request goes through the service mesh (think of it as a security checkpoint between every two services at Uber). The RL Client sitting in that mesh starts counting…
Notification Service ──5000 req/s──▶ [RL Client in Mesh] ──▶ Trip Service Limit configured: 1,000 req/s Currently seeing: 5,000 req/s ← 5× over!
2
📊
Tier 1: RL Client counts & reports
Mesh Layer
The RL Client (Tier 1) lives inside the service mesh sidecar. Every few seconds it:
• Counts how many requests it's seen
• Sends that number UP to the Zone Aggregator

Right now it doesn't block anything yet — it's still using its last known limit from the controller. It reports: "I'm seeing 5,000 req/s in zone us-east-1a"
[RL Client] counts 5,000 req/s Reports every ~1 second ↑ [Zone Aggregator us-east-1a] ← "seeing 5,000/s"
3
🗂️
Tier 2: Zone Aggregator adds it all up
Zone Aggregator
Uber runs many instances of the RL Client (one per pod/container). The Zone Aggregator collects from ALL of them in the zone and adds them up. Then it forwards the total up to the Regional Controller.
Inputs from all RL Clients in zone us-east-1a: RL Client pod-1 → 1,700 req/s RL Client pod-2 → 1,800 req/s RL Client pod-3 → 1,500 req/s ────────────── TOTAL reported ↑ 5,000 req/s [Zone Aggregator] →→ [Regional Controller] "Trip Service is getting 5,000 req/s. Limit is 1,000."
4
🧠
Tier 3: Regional Controller does the math
Regional Controller
The Regional Controller (the brain) receives the report and calculates:

"They're at 500% of the limit. I need to tell each RL Client to only allow 20% of traffic through — shed the other 80%."

It then pushes this new instruction back DOWN to all Zone Aggregators → RL Clients.
[Regional Controller] calculates: Configured limit : 1,000 req/s Current traffic : 5,000 req/s Overload ratio : 5× Decision: each RL Client should pass only 200 req/s and DROP the rest (80% drop rate) Pushes instruction ↓↓ to all RL Clients
5
🚫
Tier 1: RL Client enforces — drops 80%
Mesh Layer (again)
The RL Client receives the new limit and immediately starts enforcing it. For every 10 incoming requests, it lets 2 through and drops 8 right there at the mesh — they never even reach Trip Service.

This is probabilistic — it doesn't block the first 200 and reject the rest. It randomly drops 80% so no single caller is fully blocked.
Notification Service sends 10 requests: req-1 → ✅ PASS (reaches Trip Service) req-2 → ❌ DROP (blocked at mesh, 429 returned) req-3 → ❌ DROP req-4 → ✅ PASS req-5 → ❌ DROP req-6 → ❌ DROP req-7 → ❌ DROP req-8 → ❌ DROP req-9 → ✅ PASS (roughly 2 out of 10 get through) req-10 → ❌ DROP Trip Service only sees ~1,000 req/s ← safe!
✓
🎉
Trip Service stays healthy
Result
Trip Service never saw more than 1,000 req/s. It never crashed. The loop keeps running every few seconds — if the Notification bug is fixed and traffic drops back to normal, the controller recalculates and tells RL Clients to stop dropping traffic. Fully automatic.
BEFORE GRL: Notification bug → 5,000 req/s hits Trip Service → 💥 CRASH AFTER GRL: Notification bug → 5,000 req/s enters mesh → RL Client drops 80% at the edge → 1,000 req/s reaches Trip Service → ✅ Trip Service is fine And when the bug is fixed: Traffic drops → Controller recalculates → Drop rate goes back to 0% → ✅ Normal

⚡ The Full Picture in One View

🛡️
Tier 1 — RL Client (in the mesh): Counts requests, enforces the drop rate locally. Acts instantly — no round-trip needed.
📊
Tier 2 — Zone Aggregator: Collects from all RL Clients in a zone, sums them up, sends the total to the controller.
🧠
Tier 3 — Regional Controller: Sees global traffic, calculates how much to drop, pushes limits back down. Runs every ~1 second.
🎲
Probabilistic shedding: Instead of hard blocking (all-or-nothing), it randomly drops a %. Smooth degradation, not a cliff.
🔄
Feedback loop: Measure → Aggregate → Calculate → Push → Enforce → repeat every second. Fully automatic, no human needed.
