Streaming vs. Messaging

Understanding the fundamental tension between Kafka's ordering guarantees and pub-sub message queueing needs.

🔢 Apache Kafka — Stream-Oriented System Kafka guarantees message ordering. Consumers always see messages in the exact order they were written. This is a foundational semantic property of the system.
🔒

Exclusive Partition Ownership

Each partition is assigned to exactly one consumer in a group. No two consumers in the same group ever process the same partition concurrently.

📋

Sequential Read → Process → Commit

The recommended pattern: fetch one message, process it, commit offset, fetch the next. This preserves ordering but limits throughput severely.

🔄

At-Least-Once Delivery

Autocommit is disabled in most production setups. A message must be successfully processed before its offset is committed, preventing data loss on restarts.

🎲 Message Queue — Unordered System Traditional message queues are unordered, point-to-point, and asynchronous. Message ordering between any two messages has no semantic meaning.
🌐

Any Consumer Can Process Any Message

Work is distributed dynamically. Any available consumer instance picks up any available message. Throughput scales with number of consumers.

⚡

Inherently Parallel

Multiple consumers process multiple messages simultaneously. Adding more consumer instances linearly increases throughput with no partition limits.

🔀

No Head-of-Line Blocking

A slow or failed message does not block any other messages. Each message is independent and can be retried or skipped without affecting others.

Two Core Growing Pains

After 5 years of scaling from 1M → 12M msgs/sec, these two problems became critical blockers.

🧱 Problem 1: Partition Scalability

With 1-second RPC latency to external payment providers like Visa, achieving 1,000 events/sec requires 1,000 Kafka partitions. A cluster caps at ~200k partitions, limiting it to ~200 topics.

⚙️ The Underutilization Problem A partition can sustain 10MB/s throughput (~10,000 msgs/sec at 1KB). But with 1s processing latency, each partition handles only 1 msg/sec — a 10,000× waste of partition capacity. Reducing RPC latency to 100ms only brings it to 10 msgs/sec per partition — still extremely wasteful.

🚧 Problem 2: Head-of-Line Blocking

Kafka's sequential consumer pattern cannot handle two failure modes that are common in real-world async queuing workloads.

6-Step Processing Lifecycle

Fetch Messages from Kafka

Consumer Proxy uses the native Kafka binary protocol to fetch batches of messages. A small number of Proxy nodes consume partitions — decoupled from the service fleet size.

Push to Consumer Service (gRPC)

Each message is pushed separately to an available consumer service instance via gRPC. Multiple messages are dispatched in parallel to any number of service instances — breaking the 1-partition-1-consumer throughput ceiling.

Consumer Service Processes Message

The service handles all business logic (e.g., charge Visa, charge Mastercard, update database). It only needs to expose a gRPC endpoint — zero Kafka SDK required.

Receive gRPC Status Code

The service responds with a gRPC status code: OK (success), retriable error (back-pressure), or DLQ instruction (non-retriable error). This drives all subsequent proxy behavior.

Aggregate Results in Commit Tracker

Consumer Proxy tracks acknowledged, negative-acknowledged (DLQ), and in-flight offsets in the out-of-order commit tracker. No single slow message blocks others from being tracked.

Commit Safe Offsets to Kafka

Once a contiguous range of messages are all acknowledged or DLQ'd, Consumer Proxy commits the highest safe offset back to Kafka. This enables the next batch fetch without risk of data loss.

Why Proxy Over Client Library?

🌐

Polyglot by Default

Uber runs Go, Java, Python, and NodeJS services. A complex Kafka SDK in each language is expensive to maintain. Consumer Proxy exposes a single gRPC endpoint that any language can call.

🔧

Independent Upgrades

With 1000+ microservices, Kafka client version upgrades take months of coordination. The proxy team can redeploy the proxy at any time without touching any service codebase.

⚖️

Rebalance Storm Isolation

Rolling restarts across hundreds of service instances cause Kafka consumer group rebalance storms. Consumer Proxy isolates rebalancing to its small fleet — services are unaffected.

📊

Even Load Distribution

With 4 partitions and 40 instances using native Kafka, only 4 instances get load. Consumer Proxy fans out across all 40 instances, eliminating hotspots.

Partition Parallel Processing

Solving the partition scalability problem by decoupling consumer count from partition count.

💡 The Core Insight A single Consumer Proxy node consumes one Kafka partition. But that node can fan-out and push messages to N consumer service instances simultaneously. The partition count no longer limits processing throughput.

❌ Before: Native Kafka Consumer

Throughput is strictly bounded by partition count. To get 1000 msg/sec with 1s RPC latency, you need 1000 partitions. Partition count = consumer throughput ceiling.

throughput = partitions × (1 / rpc_latency)
1000 msg/s = 1000 partitions × (1 / 1s)

✅ After: Consumer Proxy

One partition feeds one Consumer Proxy node. That node fans out to unlimited service instances. Throughput scales with service instance count, not partition count.

throughput = partitions × (N_instances / rpc_latency)
1000 msg/s = 1 partition × (1000 / 1s)

Dead Letter Queue (DLQ)

How Consumer Proxy handles poison pills without stalling the entire pipeline.

🧨

What is a Poison Pill?

A message that cannot be processed due to a non-transient error, or takes an extremely long time (10+ minutes). In sequential consumers, it permanently stalls everything behind it.

📨

DLQ Routing

The consumer service returns a specific gRPC error code. Consumer Proxy writes the message to a dedicated DLQ Kafka topic, and marks the offset as "negative-acknowledged" in the commit tracker.

✅

Pipeline Unblocked

Negative-acknowledged offsets count as resolved in the commit tracker. Once a contiguous range of acked + dlq'd offsets is complete, Consumer Proxy commits and fetches the next batch.

DLQ Operations

🔄 Merge DLQ (Replay)

Replay poison pill messages back to the consumer service after fixing the root cause. Consumer Proxy supports bounded Kafka consumption to re-deliver only the DLQ messages.

🗑️ Purge DLQ (Discard)

Permanently discard messages that can never be processed. Used when data is stale, malformed beyond repair, or intentionally skippable.

⚠️ Why DLQ Is Essential (Not Optional) Without DLQ, a single poison pill permanently fills up the out-of-order commit tracker. New messages can never be committed because the tracker can't advance past the unresolved poison pill. Consumer Proxy stops fetching. The entire consumer pipeline grinds to a halt.
07 /

Flow Control

Preventing consumer services from being overwhelmed or starved by intelligent push rate management via gRPC status codes.

gRPC OK

Message processed successfully. Proxy marks as acknowledged. If below capacity, push rate may increase.

gRPC RESOURCE_EXHAUSTED

Consumer is overloaded. Proxy backs off using exponential backoff. Message is retried, not lost.

gRPC UNAVAILABLE

Transient service error. Proxy retries with backoff without advancing the committed offset.

gRPC DATA_LOSS / custom

Non-transient error. Proxy routes message to DLQ and marks offset as negative-acknowledged.

🎛️

Configurable Tracker Window

Operators set the max size of the out-of-order commit tracker and per-message timeout on onboarding. These define the "window" of simultaneously in-flight messages.

🔌

Circuit Breaker

If the consumer service goes completely down, the circuit breaker trips and Consumer Proxy stops pushing. This prevents a total-outage scenario from flooding the DLQ with good messages.

🔮 Planned

Adaptive Flow Control

Upcoming: TCP-inspired congestion control (AIMD) that automatically tunes push rates based on observed response times and error rates. Eliminates manual configuration.

Next Steps

Planned improvements to make Consumer Proxy even more self-managing and infrastructure-flexible.

🎛️ Adaptive Flow Control Mechanisms

▼

Manually tuning the commit tracker window size and processing timeouts is non-trivial. Small values limit throughput; large values overwhelm consumer services during spikes. Getting these numbers right requires deep knowledge of each service's behavior.

The team plans to implement Adaptive Flow Control inspired by TCP's congestion control algorithms. The proxy would dynamically adjust push rates based on observed service response times, error rates, and backpressure signals — no manual configuration required.

This mirrors TCP's AIMD (Additive Increase, Multiplicative Decrease): increase push rate gradually when things are good; cut back sharply when congestion signals appear. The result: optimal throughput automatically, even as service behavior changes over time.

🌐 Kafka Cluster Federation

▼

Consumer Proxy centralizes control over how consumer services interact with Kafka. This creates an opportunity: because services only talk to Consumer Proxy via gRPC (not directly to Kafka), the proxy can transparently redirect them to a different underlying Kafka cluster.

With Cluster Federation, the Kafka infrastructure team can move a topic between clusters — for capacity balancing, cluster deprecation, or geographic migration — without any disruption to consumer services. Services keep calling the same gRPC endpoint; the proxy handles the cluster switch seamlessly underneath.

This is a major operational improvement at Uber's scale, where managing the lifecycle of dozens of Kafka clusters is a constant challenge.

🏆 Results at Uber Consumer Proxy became the primary async queuing solution at Uber. Hundreds of services are onboarded. It improves developer velocity by hiding all Kafka internals behind a simple gRPC endpoint, while giving the Kafka infrastructure team full centralized control over upgrades, configuration, and operational management — without requiring any coordination with service teams.
